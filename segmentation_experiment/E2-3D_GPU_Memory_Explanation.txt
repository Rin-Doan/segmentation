WHY E2-3D USES LESS GPU MEMORY THAN E1 AND E2-2.5D
================================================================================

Summary of GPU Memory Usage:

E2-3D uses approximately 21% less GPU memory (0.294 GB vs 0.373 GB) despite 
processing 3D volumes instead of 2D slices.

================================================================================
PRIMARY REASON: BATCH SIZE DIFFERENCE
================================================================================

The most significant factor is the batch size configuration:

1. E1-no / E1-augmentation / E2-2.5D:
   - Batch Size: 64
   - Input: 2D slices (individual CT scan slices)
   - GPU Memory per batch: 64 slices × memory per slice
   - These models process 64 2D slices simultaneously in GPU memory

2. E2-3D:
   - Batch Size: 1
   - Input: 3D volumes (entire CT scan volumes)
   - GPU Memory per batch: 1 volume × memory per volume
   - This model processes only 1 3D volume at a time in GPU memory

Even though a single 3D volume contains more data than a single 2D slice, 
processing only 1 volume vs 64 slices results in less total data being held 
in GPU memory simultaneously.

================================================================================
MATHEMATICAL EXPLANATION
================================================================================

GPU Memory Usage = (Batch Size) × (Memory per Sample) × (Model Memory)

For E1/E2-2.5D (2D models):
- Batch Size: 64
- Memory per 2D slice: ~M_slice
- Total batch memory: 64 × M_slice
- Model weights: ~W_2D
- Total GPU Memory ≈ 64 × M_slice + W_2D

For E2-3D (3D model):
- Batch Size: 1
- Memory per 3D volume: ~M_volume
- Total batch memory: 1 × M_volume
- Model weights: ~W_3D
- Total GPU Memory ≈ 1 × M_volume + W_3D

Key Insight:
- While M_volume > M_slice (a 3D volume is larger than a single slice)
- The relationship is: M_volume << 64 × M_slice
- A typical 3D volume might be 50-200 slices, but processing 64 slices at once
  means 64× the memory footprint compared to processing 1 volume at a time

Example Calculation:
- If a 2D slice requires ~0.005 GB
- 64 slices × 0.005 GB = 0.32 GB (batch data)
- If a 3D volume (100 slices) requires ~0.05 GB
- 1 volume × 0.05 GB = 0.05 GB (batch data)
- The 3D model processes less data per batch despite each sample being larger

3. Memory Management:
   - Processing one volume at a time allows for more predictable memory usage
   - Less memory fragmentation
   - Better memory reuse patterns

4. Data Processing:
   - E2-3D processes entire volumes, which may be more memory-efficient than
     loading and processing 64 individual slices
   - Less overhead from batch preparation and data loading

================================================================================
WHY BATCH SIZE = 1 FOR E2-3D?
================================================================================

The E2-3D model uses batch_size=1 because:

1. 3D volumes are inherently large:
   - A typical CT volume can be 512×512×100-200 voxels
   - Processing multiple 3D volumes simultaneously would exceed GPU memory limits
   - Batch size of 1 is a practical constraint for 3D medical imaging

2. Memory constraints:
   - 3D convolutions require significantly more memory than 2D convolutions
   - The memory requirement grows cubically with spatial dimensions
   - Even a batch size of 2-4 might cause out-of-memory errors

3. Training stability:
   - For 3D volumes, batch_size=1 is common in medical imaging
   - Gradient accumulation can be used if larger effective batch sizes are needed
   - The model still trains effectively with batch_size=1
